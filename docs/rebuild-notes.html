Site rehaul — working notes

This document tracks direction for modernizing artisti7607.com alongside the static GitHub Pages design kit. It is not a full specification; it will grow as new static pages are added under this repo.

Implementation stack

The WordPress front-end rebuild is based on Blocksy Pro: the Blocksy parent theme plus a Blocksy child theme (see the application repo for Composer packages, Companion Pro, and theme paths). Static mockups in this design repo inform layout and chrome; they are not drop-in Blocksy templates, but they should stay compatible with Blocksy’s header/footer and global style capabilities.

Problem statement

The live site mixes navigation patterns: different headers for contexts, extended menus for some user levels, and behavior that is hard to predict when moving between public pages, logged-in areas, and tools. That increases cognitive load and makes the brand feel fragmented.

Chrome contract (all roles)

Every front-end view should share the same layout regions, whether the user is a guest, a member, or staff. Only labels and targets inside those regions change:

  • Logo — always links to the site home (not a role-specific landing unless product explicitly requires it).
  • Primary navigation — exactly five top-level destinations. The fifth slot is the “variable” column (e.g. Servizi, Area riservata, Strumenti) so marketing, product, and ops can agree on one reserved slot instead of growing new top-level rows per audience.
  • Utilities — exactly four regions: FAQ, database search, language, session. Session may bundle a small role badge with the account or sign-out control so the bar does not sprawl.
  • Footer — copyright plus four links in one row (legal + one contextual link such as Iscriviti, Guida, or Note legali). Same HTML structure on every template.

WordPress / implementation hints

  • Blocksy Pro — Prefer Blocksy’s header builder, footer builder, and global styles where they match the agreed chrome contract; extend in the child theme when mockups need markup or behavior Blocksy does not expose out of the box.
  • Implement the chrome once in the child theme (header/footer partials), not per-page PHP branches that duplicate markup.
  • Drive link sets from menus or options registered in WordPress, with validation so editors cannot exceed the agreed counts (or overflow is intentionally folded into the fifth primary or utility session).
  • Keep is_home, is_admin (bar on front), and role checks to swap URLs and labels only, not to inject a second navigation system.
  • Prefer one responsive breakpoint strategy (e.g. hamburger that still exposes the same information architecture) documented here when mobile mockups exist.
